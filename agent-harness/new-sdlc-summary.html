The Big Idea

The Shift from Syntax to Intent

"The most profound shift in software engineering isn't a new language, framework, or cloud service. It's the transition from writing code to expressing intent, and trusting intelligent systems to translate that intent into working software."

For decades, developers translated human intent into machine-readable syntax. That translation layer is collapsing. AI evolved from autocomplete (~2021) → inline suggestions (~2022) → chat-based generation (~2023) → coding agents (~2024–25) → fully autonomous agents (~2025–26) that clone repos, plan multi-file changes, run tests, and submit PRs without a human typing a single line.

The implications are profound but uneven. Implementation that once took weeks now takes hours. Requirements, architecture, and verification remain stubbornly human-paced. The result is not a faster version of the old SDLC — it is a fundamentally different workflow.

The two verification mechanisms that separate agentic engineering from vibe coding: Tests verify deterministic parts. Evaluations (evals) verify non-deterministic parts — did the agent take the right trajectory of steps, choose the right tools, and meet the quality bar? Without both, the practice is always vibe coding, regardless of prompt sophistication.

The Real Skill

Context Engineering

The quality of AI-generated code depends less on the cleverness of your prompts and more on the quality of the context provided. The question isn't "how do I trick the AI?" — it's "what would a new team member need to know to contribute effectively, and how do I encode that knowledge in a form the AI can use?"

Six Types of Agent Context

Instructions — role, goals, boundaries Knowledge — docs, diagrams, domain data Memory — session logs & persistent state Examples — few-shot demos, reference patterns Tools — APIs, scripts, external services Guardrails — hard constraints, safety rules

Static vs. Dynamic Context

Static Context (always loaded)

  • System instructions
  • Rule files (AGENTS.md, CLAUDE.md, GEMINI.md)
  • Global memory & persona definitions
  • Core guardrails

Expensive but reliable. The agent never forgets these.

Dynamic Context (loaded on demand)

  • Agent Skills (triggered by task match)
  • Tool results (retrieved during execution)
  • RAG-retrieved documents
  • Windowed session history

Efficient and scalable. Pay only for what you use.

The design decision of what belongs in static versus dynamic context is a genuine engineering trade-off — reviewed in pull requests and versioned like code. Too much static context wastes tokens and dilutes signals. Too little means the agent forgets critical rules.

The Factory Model

The developer's primary output is not code — it's the system that produces code. A factory manager designs the assembly line; the modern developer designs the development system and ensures its output meets the required standard.

What Surrounds the Model

Harness Engineering

Agent = Model + Harness

The model is the raw engine. The harness — prompts, tools, context policies, hooks, sandboxes, sub-agents, observability — is the car, the road, and the traffic laws. On Terminal Bench 2.0, one team moved from outside Top 30 to Top 5 by changing only the harness. A separate study raised a coding agent's score 13.7 points by tweaking only the system prompt, tools, and middleware — no model change at all.

Most agent failures, examined honestly, are configuration failures — a missing tool, a vague rule, an absent guardrail, a context window stuffed with noise.

Instructions & Rule Files

AGENTS.md, CLAUDE.md, GEMINI.md, skill files. Defines who the agent is and what it's forbidden from doing.

Tools

Functions, MCP servers, and APIs the agent can call, plus prose that tells the model when and how to call them.

Sandboxes & Execution Environments

Where the agent's code actually runs, what it has access to, what it cannot reach.

Orchestration Logic

Sub-agent spawning, model routing, hand-offs between specialists, and rules governing when each fires.

Guardrails & Hooks

Deterministic code at lifecycle points — before tool calls, after file edits, before commits. Things the agent should never forget but often does.

Observability

Logs, traces, evaluations, cost and latency metering. Without observability, there is no way to tell if the agent is drifting.

The Developer's Evolving Role

Conductors and Orchestrators

Conductor Mode

  • Real-time, in-IDE, keystroke-level control
  • Immediate feedback, single-file scope
  • Tools: Copilot, Cursor, Windsurf, Gemini Code Assist
  • Best for: exploratory coding, debugging, unfamiliar codebases

Orchestrator Mode

  • Asynchronous, high-level, multi-agent
  • Goal-level control, multi-file scope
  • Tools: Claude Code, Google Jules, GitHub Copilot agent mode
  • Best for: feature implementations, migrations, test generation

The 80% Problem

AI agents rapidly generate ~80% of a feature's code. The remaining 20% — edge cases, error handling, integration points, subtle correctness requirements — demands deep contextual knowledge that current models often lack. AI errors have evolved from syntax mistakes to more insidious conceptual failures that "look right" and may even pass basic tests.

Economics

The Economics of AI Development

The critical metric for engineering leaders is Total Cost of Ownership (TCO). In the AI era, OpEx is heavily dictated by the token economy. At the crossover point, vibe coding costs 3–10x more per feature in the long run.

Vibe Coding — Low CapEx, High OpEx

  • Token burn: expensive prompting loops, low first-pass success
  • Maintenance tax: AI-generated spaghetti code is days to debug
  • Security remediation: rapid code = rapid vulnerabilities

Agentic Engineering — High CapEx, Low OpEx

  • Upfront: API schemas, test suites, structured context
  • Low marginal cost per feature over time
  • Output is structurally sound, pre-tested, aligned to standards

Three Financial Levers

Context engineering — high-signal payloads, fewer retries Dynamic context + skills — pay only for what you use Intelligent model routing — large models for complex tasks, small models for test gen / CI
Actionable Guidance

Where to Start

For Individual Developers

  1. Set up AGENTS.md. Start with ten lines: stack, conventions, hard rules, workflow.

  2. Install Agent Skills for build, evaluate, deploy, and optimize workflows.

  3. Pick one repetitive workflow and make it your first agent. Building one end-to-end teaches more than reading about a hundred.

  4. Write tests and evals before generating the code. They communicate intent more precisely than any prompt.

  5. Review every line the agent produces that is going to ship. Be skeptical of anything that looks clever.

  6. Maintain your developer skills. Treat AI as a way to apply expertise at greater scale, not a substitute for it.

For Engineering Leaders

  1. Make context engineering a first-class practice. Treat AGENTS.md, eval suites, and skill libraries as code — reviewed in PRs, owned by named engineers.

  2. Set the bar at the eval, not the demo. Require eval coverage with explicit rubrics as a precondition for any agent shipping into a shared workflow.

  3. Re-shape code review for AI-generated code. Train reviewers on failure modes: hallucinated dependencies, inadequate error handling, subtle correctness gaps.

  4. Distinguish prototyping from production in team norms. Teams that keep this blurry produce prototypes that ship by accident.

  5. Invest in the harness as a shared team asset. Build it once, refine it many times.

For Organizations

  1. Treat AI-assisted development as an engineering investment. Speed without eval coverage and observability compounds into technical debt faster than any team can pay it down.

  2. Invest in the production substrate before scale. Trajectory evals in CI, traces of every agent run, scoped permissions per agent — build this before the first production agent ships.

  3. Adopt open standards. MCP for tool access and A2A for cross-agent delegation keep vendor options open and avoid re-platforming later.

  4. Plan for hybrid teams. Humans set direction, agents do implementation, clear handoff protocols govern the boundary.

  5. Reframe hiring around judgment. The most valuable engineers will be the ones who can direct agents well.

Conclusion

Intent as the New Interface

1
Structure scales, vibes don't.

For production software, specifications, tests, guardrails, and human oversight of architecture are not optional. The gap between "it seems to work" and "it works correctly under all conditions" is where production outages live.

2
AI amplifies your engineering culture.

Organizations with strong testing practices, clear architectural standards, and healthy code review get dramatically more value. AI is a force multiplier — and it multiplies both your strengths and your weaknesses.

3
The human role is evolving, not diminishing.

Skills shift from implementation to judgment, from writing code to designing the systems that produce code. The builders who understand architecture and can evaluate output critically are more valuable than ever.

"Generation is solved. Verification, judgment, and direction are the new craft."
Open Questions

What the Whitepaper Left Thin

Four things the paper names as essential but doesn't show you how to build. Each is worked through in full in the companion open-questions Q&A →

1
Testing — how do you stop an agent from gaming its own tests?

Separate the test-author from the code-author, prove the tests fail first (red), then lock them read-only so the agent can't quietly weaken them.

2
Evals — how do you actually set one up?

An eval is just a dataset + a task + a scorer. Start with 10–20 cases drawn from real failures; prefer deterministic code-based scorers and reserve LM-judges (with a written rubric) for genuinely subjective calls.

3
Context — how do the six context types get loaded?

Decide what's always-on vs. fetched-on-demand: universal rules in AGENTS.md, situational know-how in skills (triggered when the model reads their description), and large knowledge pulled in via RAG.

4
Orchestration — how do hand-offs and routing work?

Sub-agents run in isolated contexts and return summaries; specialists hand off through shared artifacts (A2A between separate services, MCP for tools). Route rule-based first, and keep the control flow deterministic.
