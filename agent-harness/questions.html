Question 1
How do you create tests in an agentic workflow — and how do you stop the agent from just making the tests pass rather than actually solving the problem?
TestingEvals

The paper says tests should be written before code, and a good eval suite "communicates intent more precisely than any natural-language prompt." But it's thin on mechanics. Do you write the tests yourself first and then hand them to the agent? Can the agent write the tests too, and if so, what stops it from writing tests that trivially pass its own output?

Also: what does the TDD loop look like concretely when an agent is doing both the test writing and the implementation?

Answer

Separate the author of the tests from the author of the code. A test-writing pass (a dedicated agent, or you) writes the tests; the coding agent only implements and never edits them. The contractor doesn't get to rewrite the contract.

But separation alone isn't enough — it just moves the gaming up a level. Three safeguards make it real:

  1. Review gate: someone confirms the tests assert the requirement, not the implementation's assumptions. Agent-written tests still need review or a spec to check against.
  2. Red-first: run tests against empty/stub code and confirm they fail. A test that passes before any code exists is vacuous. Cheapest check that catches gamed tests.
  3. Enforced read-only: "don't edit tests" is a request; a write-permission boundary is a wall. Under pressure to pass, agents will eventually edit a stubborn test. Enforce at the harness level, or feed the coding agent only pass/fail output, never the test source.

Loop: write failing tests → review + confirm red → lock → agent implements until green → review code quality.

Question 2
How do you actually set up evals — what tool do you use, and what does a minimal eval look like in practice?
EvalsTooling

The paper is light on the mechanics here. Key takeaway from the whitepaper and blog: "set the bar at the eval, not the demo" — a passing eval suite proves an agent succeeds reliably, not just once. But what does that actually look like to set up?

What tools are commonly used (LangSmith, Braintrust, custom scripts)? What's the minimum viable eval for a new agent? What do scoring rubrics look like, and what makes a good LM judge?

Answer

An eval is just three pieces: a dataset, a task, and a scorer. A dataset of inputs (optionally with reference answers), a task that runs your agent on each one, and one or more scorers that return a number from 0 to 1. Tools like Braintrust or LangSmith are a convenience layer around that loop — a plain for loop with asserts is already a legitimate eval. The whole thing in Braintrust:

from braintrust import Eval
from autoevals import Factuality

Eval(
    "support-agent",                      # project
    data=lambda: [                        # 1. dataset: input + reference
        {"input": "Reset my password", "expected": "Sends a reset link"},
        {"input": "Cancel my plan",    "expected": "Routes to billing"},
    ],
    task=lambda q: my_agent(q),           # 2. the agent under test
    scores=[Factuality],                  # 3. scorer(s), each returns 0..1
)

Three things turn that skeleton into something that actually holds the line:

  1. Start tiny, from real failures. The minimum viable eval is 10–20 hand-written cases drawn from bugs you've actually seen — not 1,000 synthetic ones. Each case is an input plus what "correct" means. Run it on every change; a drop is a red bar. This is "set the bar at the eval, not the demo" in practice — the suite proves the agent succeeds reliably, not that it worked once.
  2. Prefer code-based scorers; reserve the LM judge for the fuzzy stuff. If you can check it deterministically — exact match, regex, valid JSON schema, "did it call the right tool," "did the tests pass," "is the number right" — do that. It's free, fast, and not gameable. Only reach for an LM judge when correctness is genuinely subjective: tone, helpfulness, "did it actually answer the question."
  3. A good LM judge has a written rubric, not a vibe. Give it discrete grades mapped to scores (A / B / C → 1 / 0.5 / 0), not "rate 1–10" — models can't reliably tell a 6 from a 7. Make it reason before it grades (chain-of-thought), pin the judge model so scores don't drift, and calibrate against a handful of human-labeled examples. Pairwise ("is A or B better?") beats absolute scoring. And grade against the requirement, not the model's own assumptions.

Tooling: start with a script. Graduate to Braintrust or LangSmith when you need to track scores across runs, diff regressions, store datasets, and share results — i.e. when the eval becomes a team asset instead of a one-off check. (Also in the space: OpenAI Evals, Inspect, promptfoo.)

Question 3
How do you get the six types of context into an agent — and when should each one be loaded?
Context EngineeringAgent Skills

The six types are: Instructions, Knowledge, Memory, Examples, Tools, and Guardrails. The paper distinguishes static context (always loaded) from dynamic context (loaded on demand via skills/RAG), but doesn't show how you wire that up.

Concretely: how do you configure what goes in AGENTS.md vs. what lives in a skill file? How do skill files get triggered — is it keyword matching, LLM decision, or explicit invocation? How does RAG fit in for the Knowledge type?

Answer

Getting context in is easy; the real decision is what's always-on vs. fetched-on-demand. Your context window is a budget — spend it on what every turn needs and lazy-load the rest. Each of the six types maps to a mechanism and a load policy:

Type Where it lives / how it's wired Loaded
InstructionsAGENTS.md / CLAUDE.md, system promptStatic — always
GuardrailsSystem-prompt rules + harness-enforced permissions / hooksStatic — always & enforced
ToolsFunction/tool schemas, MCP serversStatic registration (can defer-discover)
ExamplesFew-shot in instructions, or pulled from skills / retrievalEither — static if core, else fetched
KnowledgeRAG / vector store, MCP resources, skill reference filesDynamic — on demand
MemoryPersistent store / files across sessionsHybrid — index static, recall on demand

The three wiring questions that actually trip people up:

  1. AGENTS.md vs. a skill — size and universality decide. AGENTS.md (the open, cross-tool standard; CLAUDE.md is Claude Code's flavor) holds the small, always-true stuff every task needs: project conventions, build/test commands, guardrails. If it's large, situational, or only relevant to some tasks, it goes in a skill so it doesn't tax every turn. Rule of thumb: if you wouldn't want it in the system prompt for every request, it's a skill — not AGENTS.md.
  2. Skills trigger by model decision, not keyword match. A skill is a folder with a SKILL.md whose YAML frontmatter has a name + description. Only that metadata is always loaded (a cheap index in the system prompt); the model reads the description, decides the skill is relevant, then pulls the full body from the filesystem — "progressive disclosure." So the description is the trigger: write it as "what it does and when to use it." Three loading levels: metadata (always) → SKILL.md body (on trigger) → bundled scripts / reference files (on demand, only their output costs tokens). Skills can also be invoked explicitly (e.g., a slash command), but model-decided is the default.
  3. RAG is how the Knowledge type scales past the window. Embed your docs / wiki / code into a vector store; at query time retrieve the top-k relevant chunks and inject them. You never load the whole corpus — just what this query needs. For a small, stable knowledge base, a skill reference file or an MCP resource is often simpler than standing up retrieval.

Default everything to dynamic; promote to static only when it earns it. Static context is fast and reliable but permanently taxes every turn, and the window is finite. Keep the always-on layer lean — instructions, guardrails, core tools — and load knowledge, niche examples, and specialist skills on demand. Treat that static/dynamic boundary as architecture and version it alongside the code.

Question 4
How does orchestration logic actually work — sub-agent spawning, model routing, hand-offs?
OrchestrationMulti-agent

The paper names orchestration logic as a core harness component ("sub-agent spawning, model routing, hand-offs between specialists, and the rules that govern when each one fires") but doesn't explain how you build or configure it.

Questions: How does a planning agent hand off to a coding agent? What protocol governs that (A2A? shared session state? MCP?)? How do you do intelligent model routing in practice — is it rule-based or does a meta-agent decide? What does a minimal orchestration setup look like vs. a production one?

Answer

Orchestration is three decisions — who runs next, with which model, and how the baton gets passed — and you want as much of it in deterministic code as you can get away with. Letting one orchestrator-LLM decide everything is flexible but unpredictable; production setups are mostly a hard-coded control flow (a graph or pipeline) with model judgment only at the nodes.

  1. Spawning: a sub-agent is a fresh context window with a scoped job. The orchestrator calls a sub-agent tool with a focused prompt and a subset of tools; the sub-agent works in its own isolated context and returns only a result/summary — not its whole transcript. That isolation is the point: the lead stays clean and you can fan out several sub-agents in parallel. Caveat: multi-agent fan-out burns roughly 15× the tokens of a single-agent chat (Anthropic's multi-agent research figure), so reserve it for high-value, parallelizable, breadth-first work — it's not the default.
  2. Hand-offs: pass an artifact, not a conversation. The robust planner→coder pattern is for the planner to emit a structured artifact — a plan file, a spec, a task list — that the coder reads. The hand-off is shared state on disk, not a live wire. Two open protocols formalize this when agents are separate services:
    • A2A (Agent2Agent) — horizontal, agent↔agent. An agent publishes an Agent Card at /.well-known/agent-card.json advertising its skills and endpoints; a client agent discovers it and delegates a task over JSON-RPC.
    • MCP (Model Context Protocol) — vertical, agent↔tools/data. It's how an agent reaches tools, not how agents hand off to each other.
    They compose: an A2A client delegates to an A2A agent, which then uses MCP to reach its tools. But for sub-agents inside one harness you need neither — a sub-agent call plus a shared file is the whole hand-off. Reach for A2A only when the agents are independent services (different teams, vendors, or processes).
  3. Routing: start rule-based, role → model. Map each role to a model by difficulty: a cheap/fast model (e.g., Haiku) for triage, classification, and narrow sub-tasks; a frontier model (e.g., Opus) for orchestration and hard reasoning or coding. That's static config and covers most cases. A meta-agent router — a small model that scores difficulty and dispatches — is a production optimization: more adaptive, but it adds latency, cost, and another failure point. Don't build it until a fixed role→model mapping is demonstrably leaving quality or cost on the table.

Minimal vs. production. Minimal is one orchestrator with a sub-agent/Task tool: it decides when to spawn a focused sub-agent, gets the result back, and hands off via prompt + a shared file — no framework. Production is an explicit graph/state-machine (LangGraph, Google ADK, CrewAI, the Claude or OpenAI agent SDKs) with defined roles, scoped tools, per-role model routing, persistent shared state, tracing, retries/guardrails, and an eval suite over the whole pipeline. The jump isn't more agents — it's making the control flow deterministic and observable.
