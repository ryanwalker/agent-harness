Key insight Most agent failures, examined honestly, are configuration failures — not model failures. Fix the harness before changing the model.

Five Parts of Every Agent

Model

Reasoning engine. Reads context, decides next step, produces output.

Tools

Connect model to the world: APIs, code execution, databases, other agents.

Memory

State: past interactions, project rules, context across sessions.

Orchestration

The loop: assembles context, dispatches tool calls, captures results, decides whether to continue.

Deployment

Hosting, identity, observability, production infrastructure.

Harness Components (What Surrounds the Model)

  1. Instructions / Rule Files — AGENTS.md, CLAUDE.md, GEMINI.md, skill files, subagent prompts. Versioned in source control.
  2. Tools — functions, MCP servers, APIs the agent can call + prose for when and how to call them.
  3. Sandboxes / Execution Environments — what the agent can reach and what it cannot.
  4. Orchestration Logic — subagent spawning, model routing, handoffs between specialists.
  5. Guardrails / Hooks — deterministic code at lifecycle points (pre-tool-call, post-file-edit, pre-commit). Things the agent should never forget but often does.
  6. Observability — logs, traces, evals, cost and latency metering. Without this there is no way to tell whether the agent is quietly drifting.

The Vibe Coding → Agentic Engineering Spectrum

Position on this spectrum is determined by verification rigor, not which tool you use. The same coding agent can be used at either end.

🎲 Vibe Coding

  • Casual prompts
  • "Does it seem to work?"
  • No rule files
  • No automated evals
  • Risk: HIGH
  • Right for: prototypes, hackathons

⚙️ Structured AI-Assisted

  • Detailed prompts + constraints
  • Manual testing + spot-checks
  • Informal rule files
  • Selective review
  • Risk: MODERATE
  • Right for: features in established codebases

🏭 Agentic Engineering

  • Formal specs + architecture docs
  • Automated evals + CI/CD gates
  • Versioned, PR-reviewed rule files
  • Comprehensive review
  • Risk: LOW
  • Right for: production systems at scale

Economics: Why Vibe Coding Loses at Scale

Vibe Coding — Low CapEx, High OpEx

  • Token burn from unstructured prompting loops
  • Maintenance tax: structural inconsistency compounds
  • Security remediation
  • Context collapse on large tasks
  • Crossover: 3–10× more expensive per feature at scale

Agentic Engineering — High CapEx, Low OpEx

  • Upfront: API schemas, eval suites, structured context
  • Marginal cost per feature drops sharply after setup
  • Dense rule files increase first-pass success rate
  • Intelligent model routing keeps ongoing costs down
  • Compounds across projects over time

Static vs. Dynamic Context Split

This boundary is a first-class architectural decision. Treat it like code — version it, review it in PRs, assign ownership.
STATIC — loads every turn

Always-On Context

  • System instructions
  • Rule files (AGENTS.md, CLAUDE.md)
  • Core guardrails
  • Global memory

⚠ Token cost: high. Keep lean (~60 lines/file max).

DYNAMIC — loads on demand

On-Demand Context

  • Agent Skills (triggered by task match)
  • Tool results (retrieved during execution)
  • RAG documents
  • Windowed session history

✓ Pay only when needed. Heavy references go here.

Developer Modes

🎹 Conductor Mode

  • Real-time, in-IDE, keystroke-level control
  • Developer always in the loop
  • Single-file scope, immediate feedback
  • Best for: exploration, prototyping, learning new APIs
  • Risk: becomes a bottleneck if directing every keystroke

🎛 Orchestrator Mode

  • Async, goal-level delegation
  • Review outcomes, not generation
  • Multi-file scope, delayed feedback
  • Best for: migrations, test gen, refactors
  • Requires: specification + evaluation skills

Build & Rollout Playbook

Starting (Individual)

1

Create a rule file in every project

Start with 10 lines: stack version pins, architectural patterns, banned packages, test locations. Filename: AGENTS.md or CLAUDE.md.

2

Apply the Ratchet Principle from day one

Every time the agent makes a mistake, add a rule that prevents recurrence. Failures are permanent signals, not one-offs.

3

Write tests before generating code

Tests become the specification. They communicate intent more precisely than any natural-language prompt.

4

Wire hooks at lifecycle points

Pre-tool-call, post-file-edit, pre-commit. Gate destructive commands behind explicit approval.

5

Keep tool outputs off the context window

Large outputs go to disk; pass paths instead. Use compaction/summarization when context fills.

Rolling Out to a Team

1

Assign a DRI before expanding access

One person (or small team) owns the harness config. Without this, setups fragment and become tribal knowledge.

2

Governance first, breadth second

Define approved skills, code review requirements, and limited initial scope before rolling out to the team.

3

Treat rule files as production code

Require PR review. Version alongside application code. Assign named owners.

4

Distribute as plugins, not tribal knowledge

Bundle working setups and distribute org-wide. Prevents fragmentation and ensures consistency.

5

Audit CLAUDE.md every 3–6 months

Remove compensatory workarounds as model capabilities improve. Config written for older models can constrain newer ones.

6

Set the bar at the eval, not the demo

A demo proves success once. A passing eval suite proves reliable operation. Require eval coverage before any agent ships into shared workflows.

Red Flags in Agent PRs

  • Altered test assertions — check tests before implementation code
  • Lowered coverage thresholds or skipped linting
  • Untrusted input flowing into prompt calls (prompt injection surface)
  • Diffs too large to meaningfully review (agent PRs are 51% larger on average)
  • Hallucinated dependencies — verify every import is a real package
  • Changes that weaken CI gates
A human must own the merge decision. Accountability, strategic judgment, and requirement-level thinking do not delegate.

Verification — The New Engineering Craft

"Generation is solved. Verification, judgment, and direction are the new craft."
Set the bar at the eval, not the demo.

1 · Tests

Deterministic input/output coverage. Written before generation — they become the specification.

2 · Output Evaluation

Is the final result correct? Use LM-as-judge for non-deterministic outputs.

3 · Trajectory Evaluation

Was the reasoning path and tool call sequence sound? A correct output via wrong steps is a latent failure.

Tool-Agnostic Standards (Avoid Lock-In)

MCP

Model Context Protocol. Open standard for connecting tools/data to any AI assistant. Supported across tools.

A2A

Agent2Agent Protocol. Cross-agent delegation and handoff. Adopt now to avoid re-platforming later.

llms.txt

Machine-readable library documentation index. Any library publishing this can be served to any MCP client.

mcpdoc

Serves any llms.txt-indexed doc site as an MCP server. Not ADK-specific — works for any library.

AGENTS.md / CLAUDE.md

Rule file conventions supported across tools. Version in source control alongside application code.

Shared Docs MCP Setup (team rollout pattern)

{
  "mcpServers": {
    "your-library-docs": {
      "command": "uvx",
      "args": [
        "--from", "mcpdoc", "mcpdoc",
        "--urls", "LibName:https://your-lib.dev/llms.txt",
        "--transport", "stdio"
      ]
    }
  }
}

Commit as mcp.json in the project repo — every engineer's IDE gets the same documentation context automatically.

Model Routing (Direct Cost Lever)

Frontier Models

  • Complex reasoning
  • Architecture decisions
  • Unfamiliar codebases
  • Initial implementation

Smaller / Cheaper Models

  • Test generation
  • CI checks
  • Code review passes
  • Deterministic / repetitive tasks
Don't route everything to the most expensive model. Model routing is a direct financial lever.

Interview Talking Points

  • Most agent failures are harness failures. I start by auditing the configuration, not the model.
  • Verification is the high-leverage skill now. Generation is commoditized. I can demonstrate I have a review system, not just a generation workflow.
  • The 80% problem is real — I keep humans in the loop for edge cases and integration seams where models lack domain context.
  • My PR review framework: blast radius × lifespan × knowledge distribution. I tier review effort based on consequence, not uniformly apply the same process to everything.
  • Every agent mistake is a permanent signal. I apply the Ratchet Principle — the failure goes into the harness config so it can't repeat.
  • Tests written before generation are the specification, not an afterthought. They communicate intent more precisely than any natural-language prompt.
  • I use heterogeneous reviewers because no single AI reviewer catches everything — they flag different lines 93.4% of the time.
  • Context engineering is an architectural decision, not a prompt-engineering trick. I version the static/dynamic boundary alongside application code.
