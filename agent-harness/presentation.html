The Spectrum

Vibe Coding → Agentic Engineering

Position determined by verification rigor, not which tool you use.

🎲 Vibe Coding

  • Casual prompts
  • "Does it seem to work?"
  • No rule files
  • No automated evals
  • Risk: HIGH
  • Right for: prototypes

⚙️ Structured AI-Assisted

  • Detailed prompts + constraints
  • Manual testing
  • Informal rule files
  • Selective review
  • Risk: MODERATE
  • Right for: established codebases

🏭 Agentic Engineering

  • Formal specs + architecture docs
  • Automated evals + CI/CD gates
  • Versioned rule files
  • Comprehensive review
  • Risk: LOW
  • Right for: production systems
Economics

Why Vibe Coding Loses at Scale

Vibe Coding — Low CapEx, High OpEx

  • Token burn from unstructured loops
  • Maintenance tax: structural inconsistency compounds
  • Security remediation
  • Context collapse on large tasks
  • Crossover: 3–10× more expensive per feature at scale

Agentic Engineering — High CapEx, Low OpEx

  • Upfront: API schemas, eval suites, rule files
  • Marginal cost per feature drops sharply
  • Dense rule files increase first-pass success rate
  • Model routing keeps ongoing costs down
  • Compounds across projects over time
The 80% Problem
Agents reliably produce the first 80% of any feature fast.
The remaining 20% — edge cases, integration seams, subtle correctness — requires domain judgment models lack.

Most production failures originate in this 20%.
Error character has shifted: syntax mistakes → conceptual failures that look right and pass basic tests.

Do not outsource architectural decisions.

Context Architecture

Static vs. Dynamic Split

This boundary is a first-class architectural decision. Version it. Own it.

STATIC — loads every turn
  • System instructions
  • Rule files (AGENTS.md, CLAUDE.md)
  • Core guardrails
  • Global memory

⚠ Token cost: high. Keep lean (~60 lines/file max).

DYNAMIC — loads on demand
  • Agent Skills (by task match)
  • Tool results (during execution)
  • RAG documents
  • Windowed session history

✓ Pay only when needed. Heavy references go here.

Loading Pattern

Progressive Disclosure

Agent Skills: portable packages of procedural knowledge. Load only when the task calls for it.

1 · Startup

Lightweight metadata only

→

2 · Task Match

Full skill instructions load

→

3 · Execution

Deep reference material pulled on demand

Solves four problems: context rot from overloaded prompts · absence of procedural memory · multi-agent overhead · portability across tools

Developer Modes

Conductor vs. Orchestrator

🎹 Conductor Mode

  • Real-time, in-IDE, keystroke-level control
  • Developer always in the loop
  • Single-file scope, immediate feedback
  • Best for: exploration, prototyping, learning new APIs
  • Risk: bottleneck if directing every keystroke

🎛 Orchestrator Mode

  • Async, goal-level delegation
  • Review outcomes, not generation
  • Multi-file scope, delayed feedback
  • Best for: migrations, test gen, refactors
  • Requires: specification + evaluation skills

Most developers move fluidly between both in a single day.

Rollout Playbook

Starting (Individual)

1

Create a rule file in every project

Start with 10 lines: stack pins, patterns, banned packages, test locations. File: AGENTS.md or CLAUDE.md.

2

Apply the Ratchet Principle from day one

Every agent mistake → add a rule. Failures are permanent signals, not one-offs.

3

Write tests before generating code

Tests become the specification. More precise than any natural-language prompt.

4

Wire hooks at lifecycle points

Pre-tool-call, post-file-edit, pre-commit. Gate destructive commands behind explicit approval.

5

Keep tool outputs off the context window

Large outputs to disk; pass paths instead. Compaction when context fills.

Rollout Playbook

Rolling Out to a Team

1

Assign a DRI before expanding access

One person owns the harness config. Without this, setups fragment.

2

Governance first, breadth second

Approved skills list, review requirements, limited scope — before rolling out broadly.

3

Treat rule files as production code

PR review required. Versioned alongside app code. Named owners.

4

Distribute as plugins, not tribal knowledge

Bundle working setups org-wide. Prevents fragmentation.

5

Audit CLAUDE.md every 3–6 months

Remove compensatory workarounds as model capabilities improve.

6

Set the bar at the eval, not the demo

Require eval coverage before any agent ships into shared workflows.

Red Flags in Agent PRs

Check These First

Altered test assertions — check tests before implementation code
Lowered coverage thresholds or skipped linting
Untrusted input flowing into prompt calls (prompt injection surface)
Diffs too large to meaningfully review (agent PRs 51% larger on average)
Hallucinated dependencies — verify every import is a real package
Changes that weaken CI gates
A human must own the merge decision. Accountability and strategic judgment do not delegate.

Verification

The New Engineering Craft

"Generation is solved. Verification, judgment, and direction are the new craft."

1 · Tests

Deterministic input/output coverage. Written before generation — they become the specification.

2 · Output Evaluation

Is the final result correct? Use LM-as-judge for non-deterministic outputs.

3 · Trajectory Evaluation

Was the reasoning path and tool call sequence sound? A correct output via wrong steps is a latent failure.

Set the bar at the eval, not the demo. A demo proves success once. A passing eval suite proves reliable operation.
Avoid Lock-In

Tool-Agnostic Standards

MCP

Model Context Protocol. Open standard for connecting tools/data to any AI assistant.

A2A

Agent2Agent Protocol. Cross-agent delegation. Adopt now to avoid re-platforming.

llms.txt

Machine-readable library docs. Any library publishing this works with any MCP client.

mcpdoc

Serves any llms.txt site as an MCP server. Works for any library, not just ADK.

AGENTS.md

Rule file convention supported across Claude, Cursor, Gemini CLI, and others.

Shared docs setup — commit mcp.json to every project repo:

uvx mcpdoc --urls LibName:https://your-lib.dev/llms.txt --transport stdio
Cost Optimization

Model Routing

Frontier Models

  • Complex reasoning
  • Architecture decisions
  • Unfamiliar codebases
  • Initial implementation

Smaller / Cheaper Models

  • Test generation
  • CI checks
  • Code review passes
  • Deterministic / repetitive tasks
Model routing is a direct financial lever. Don't route everything to the most expensive model.
